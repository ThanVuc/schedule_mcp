# Rebuild Pipeline

Input: `LLD-Phase-2.md`

Output: `Rebuild-Pipeline.md`

## 1. Pipeline Overview

| Step | Name | Purpose |
| --- | --- | --- |
| 1 | Ingestion | Receive raw documents and preserve source boundaries |
| 2 | Classification & Extraction | Extract maximum signals with pattern-first, coverage-first logic |
| 3 | Normalization | Convert extracted signals into normalized internal items |
| 4 | Reconciliation | Merge related normalized items into canonical items |
| 5 | Canonical Model Construction | Build feature-centered canonical model |
| 6 | Task Generation | Generate sprint tasks from canonical model |

### Process (Pattern-First)
1. Receive markdown `content` as bytes from ingestion output.
2. Decode bytes to string for extraction processing.
3. Preprocess text for robust pattern detection.
4. Run window scan to recover broken multi-line patterns.
5. Run line/table/heading scans for API, DB, feature, and task signals.
6. Detect data finish: finalize and freeze detected signal set.
7. Continue with AI refinement to improve wording and add additional meaning/context.
8. Group all signals into `api`, `databases`, `tasks`, `features`.
9. Assign `signal_origin` for each signal:
  - `explicit`: directly present in source text/table/heading.
  - `derived`: generated by deterministic transform rules from explicit signals.
  - `inferred`: created by model inference when no explicit/derived signal exists.
10. Apply Rule-Based Hard Filter as final extraction gate before pushing `SignalItemDTO`.

### 3.1 Preprocess
- Normalize text for matching only: lowercase copy + keep original text for output.
- Split structure into lines, headings, tables, paragraphs.
- Optional flatten text for cross-line pattern recovery.

### 3.2 Window Scan (Critical)

Problem example:

```text
POST
/api/v1/groups
```

Window builder:

```python
WINDOW_SIZE = 3
STEP = 1

windows = []
for i in range(0, len(lines) - WINDOW_SIZE + 1, STEP):
  chunk = lines[i:i + WINDOW_SIZE]
  windows.append("\n".join(chunk))
```

Apply patterns on windows:

```python
for window in windows:
  detect_api(window)
  detect_db(window)
  detect_task(window)
```

Smart trigger (optional):

```python
if any(keyword in line.lower() for keyword in ["api", "endpoint", "method", "/", "{", "}"]):
  scan_window()
```

Rules:
- Do not hard-dedup window results.
- Window scan is for pattern recovery, not semantic merge.

### 3.3 API Signal Detection (Strong)

Core method + endpoint pattern:

```regex
(GET|POST|PUT|PATCH|DELETE|OPTIONS|HEAD)\s+(/[A-Za-z0-9_\-/{}/]+)
```

Table pattern:
- Detect table headers containing `method` and `endpoint`.

Fallback endpoint pattern:

```regex
/(api|v\d+)/[A-Za-z0-9_\-/{}/]+
```

Verb fallback mapping:
- create -> POST
- update -> PUT/PATCH
- delete -> DELETE
- get/list -> GET

API rules:
- Keep API even if method is missing.
- Keep API even if endpoint format is imperfect.
- Allow partial match.

### 3.4 Database Signal Detection (Strong)

Table structure hint:
- Column headers like `column`, `type`, constraints.

Inline schema pattern:

```regex
([a-zA-Z_][a-zA-Z0-9_]*)\s*\((.*?)\)
```

Column pattern:

```regex
([a-zA-Z_]+)\s*:\s*([A-Za-z0-9()]+)
```

Constraint keywords:
- PRIMARY KEY
- FOREIGN KEY
- NOT NULL
- UNIQUE

### 3.5 Feature Signal Detection (Medium)

Heading-based:
- Lines starting with `#`, `##`, `###`.

Keyword filter EN:
- feature, module, spec, design, flow, sequence, process, ui, screen, page, form

Keyword filter VI:
- chức năng, module, đặc tả, thiết kế, luồng, xử lý, màn hình, giao diện

Classification:
- flow/sequence -> sequence
- ui/screen/page/form -> ui
- spec/module/design -> spec
- else -> general

Cleanup:
- Remove numbering prefixes.
- Normalize spaces.

### 3.6 Task Signal Detection (Very Important)

Planning tasks (strong):
- Table rows with headers like `Task Name`, `Assignee`, `Estimation`.
- Bullet lines with action verbs.

Verb detection EN:
- implement, build, create, add, design, update, delete, integrate, setup, configure

Verb detection VI:
- xây dựng, tạo, thiết kế, triển khai, cập nhật, xoá, tích hợp, cấu hình

Regex:

```regex
^(implement|build|create|add|design|update|delete|integrate|setup|configure)\b
```

Derived tasks (from Design/Requirement):
- API -> `Implement {API}`
- Database -> `Design {database} schema`, `Create migration for {database}`
- Feature -> `Implement {feature}`

### 3.7 Priority and Strategy

Priority order:
1. Planning tasks
2. Explicit verb tasks
3. API-derived tasks
4. DB-derived tasks
5. Feature-derived tasks

Scenario strategy:
- Planning exists: use planning tasks as base, preserve 100%, augment by api/databases, ignore features for task priming.
- No Planning, Design exists: detect dominant signal.
  - API-dominant -> api + databases
  - Feature-dominant -> features
  - Mixed -> both
- Only Requirement exists: use features.

### 3.8 Cleanup and Grouping

Cleanup:
- Trim text.
- Remove empty values.
- Allow duplicates (light dedup: exact match only).

Final grouping:
- tasks <- planning + derived
- api <- API detection
- databases <- DB detection
- features <- heading + flow + UI

Core principles:
- Maximize recall over precision.
- Pattern first, AI later.
- Never drop signals.
- Accept noise.
- Use windows to recover broken patterns.

Mental model:

```text
detect -> recover (window) -> classify -> derive
```

Coverage validation:
- Planning must have task items.
- Design should have api or databases or feature items.
- Requirement must have feature items.

### 3.9 Rule-Based Hard Filter (Final Gate)

Goal:
- Keep recall high while dropping only deterministic garbage signals.

Execution point:
- Apply after detection and grouping, immediately before final extraction output.

Rule A - Empty, too short, or obvious junk:

```regex
^(.{0,2}|test|demo|example|asdf|qwerty|lorem|null|undefined)$
```

```python
if len(text.strip()) < 6:
  DROP
if junk_regex.match(text.lower()):
  DROP
```

Rule B - Non-structured API rejection:

```regex
(GET|POST|PUT|PATCH|DELETE|OPTIONS|HEAD)\s+/[^\s]+
```

```regex
/(api|v\d+)/[A-Za-z0-9_\-/{}/]+
```

```python
if signal_type == "api":
  if not match(ex_api1) and not match(ex_api2):
    DROP
```

Rule C - Broken endpoint rejection:

```regex
^/?[a-zA-Z0-9_\-]+(\s+[a-zA-Z0-9_\-]+)?$
```

Condition:
- Drop endpoint-like text that is only isolated words and not a valid path.

Rule D - DB invalid structure:

```regex
([a-zA-Z_]+)\s*:\s*([A-Za-z0-9()]+)
```

```regex
PRIMARY KEY|FOREIGN KEY|NOT NULL|UNIQUE
```

```python
if signal_type == "database":
  if not (match(ex_db1) or match(ex_db2)):
    DROP
```

Rule E - Feature generic heading noise:

```regex
^(overview|introduction|summary|general|misc|other|etc)\s*$
```

```python
if signal_type == "feature" and match(ex_feat1):
  DROP
```

Rule F - Weak verb-only task rejection:

```regex
^(implement|create|build|update|delete|fix|add|design|setup)\s*$
```

```python
if signal_type == "task" and match(ex_task1):
  DROP
```

Rule G - Noise keyword blacklist:

```regex
\b(test|testing|todo|fixme|hack|tmp|temporary|dummy|sample)\b
```

```python
if blacklist_regex.search(text.lower()):
  DROP
```

Final extraction gate logic:

```python
def extraction_filter(signal):
  if too_short_or_junk(signal.text):
    return "DROP"

  if signal.signal_type == "api" and not valid_api(signal):
    return "DROP"

  if signal.signal_type == "database" and not valid_database(signal):
    return "DROP"

  if signal.signal_type == "feature" and is_generic_feature(signal):
    return "DROP"

  if signal.signal_type == "task" and is_verb_only_task(signal):
    return "DROP"

  return "KEEP"
```

  - preserve these tasks as user-created intent throughout normalization.

### 4.1 Semantic Filter (Final Gate)

Goal:
- Remove semantically meaningless items that passed syntax-level extraction checks.

Execution point:
- Apply after embeddings and cluster metadata are available.

Step 1 - Similarity threshold outlier filter:
- Drop when cosine similarity to nearest cluster/item is less than 0.35.
- Equivalent distance-based rule can be used: distance to nearest cluster greater than 0.75.

Step 2 - Cluster isolation rule:

```python
if cluster_size == 1 and similarity_to_all < 0.4:
  return "DROP"
```

Final semantic gate logic:

```python
def semantic_filter(item):
  if cosine_to_nearest(item) < 0.35:
    return "DROP"

  if is_isolated_singleton(item):
    return "DROP"

  return "KEEP"
```

### Process
1. Filter clusters with size >= 2 for AI merge.
2. LLM Call 1 (Merge Prompt): merge per cluster and per `signal_type` only.
3. Deterministic post-processing (pattern + code) on merged/single items.
4. Preserve cluster_id and source lineage.
5. Keep single-item clusters unchanged.
6. Hard protection rule:
  - never merge or delete items where `source_file_type = Planning` and `signal_origin = explicit`.
  - keep planning explicit task items exactly as user intent, even if they look similar to other task items.

### 5.1 Merge + Deterministic Post-Processing (No Enrich LLM)

Goal:
- Keep merge decision logic in LLM (if cluster merge is needed) and move all post-merge shaping to deterministic logic.

Execution order:
1. Merge call first.
2. Run deterministic alias/title cleanup second.
3. Never run any enrichment LLM call.

LLM Call 1 - Merge Prompt (Decision Only):
- Input: items in same `cluster_id` and same `signal_type`.
- Output: one merged canonical item or explicit no-merge decision.
- Hard constraints:
  - No rewriting business meaning.
  - No fabricated fields.
  - Must preserve `signal_origin` priority: explicit > derived > inferred.
  - Must preserve all source lineage (`source_file_name`, `source_file_type`).
  - Must never merge or remove planning explicit task items.

Prompt template:

```text
You are a merge decision engine.
Task: decide whether input items represent the same canonical entity.

Rules:
- Merge only if semantic intent is equivalent.
- Keep factual meaning unchanged.
- Keep source lineage lossless.
- Prefer signal_origin in this order: explicit > derived > inferred.

Return strict JSON:
{
  "action": "MERGE" | "KEEP_SEPARATE",
  "canonical_item": { ... } | null,
  "reason": "short deterministic reason"
}
```

Deterministic post-processing (pattern + code only):
- Input: merged canonical item from Call 1 or unchanged single-item cluster output.
- Output: same schema item with deterministic alias/title cleanup.
- Allowed sources for alias generation:
  - cluster item titles
  - existing cluster aliases (if safe)
  - structural transformations of titles (no semantic expansion)
- Forbidden:
  - no LLM-generated aliases
  - no semantic expansion/inference
  - no external knowledge injection

### 5.2 Deterministic Alias Strategy

Cluster vocabulary source:
- Build normalized cluster vocabulary from all titles and aliases in the cluster.
- Do not use ID-based placeholders or semantic placeholders for matching.

Alias generator:
1. Normalize text (trim, collapse spaces, lowercase copy for matching).
2. Remove IDs/placeholders (`{id}`, UUID-like fragments, numeric-only tails).
3. Transform by type:
   - API path -> readable resource phrase (preserve method/path semantics).
   - task title -> condensed actionable noun phrase.
   - db schema title -> readable table/entity phrase.
4. Keep only aliases present in cluster surface forms or valid structural transforms.

Strict alias filter:
- Reject noise aliases (examples: `system`, `feature`, `module`, `api detailed`, `sequence diagram`).
- Reject fragmented tokens (examples: `t`, `m`, `c`, `u`).
- Reject too-short aliases (< 4 meaningful characters).
- Reject aliases outside cluster surface forms + valid transforms.

Noise filtering layer:
- Hard rule: substring blacklist/pattern match.
- Soft score:
  - short phrase penalty
  - generic prefix penalty (`system`, `feature`, `module`)
- Reject alias when score >= threshold.

Output constraints:
- deterministic across runs
- no enrichment artifacts
- alias-safe (no token explosion)
- preserve original cluster intent
- planning explicit task items are excluded from merge/removal candidates.

- Semantic alias expansion logic.
- Functions/components equivalent to:

  - `overlay_enriched_item`
  - `merge_enrichment`
  - `BuildReconciliationEnrichPrompt`

### Process

   - `features[]`
2. Apply signal priority:
   - tasks
   - apis + db_schema
   - features
3. Apply optional linking only:
   - If feature exists, try link api/db/task to feature.
   - If no feature match, keep items independent.
4. Derive tasks only if needed:
   - API -> Implement API
   - DB -> Design schema
   - Feature -> Implement feature
5. Lossless guarantee:
   - do not drop item
   - do not merge item
   - do not rewrite meaning

Notes:
- Cluster is for dedup only in reconciliation and is ignored in canonical stage.
- Canonical model is flat with optional linking, not a forced hierarchical tree.

### Generation Strategy

Case 1: Has tasks (Planning)
- Goal: full coverage tasks, no miss.
- Keep all input tasks (no drop).
- Normalize and clean descriptions deterministically.
- Do not regenerate same tasks.
- Optional infer only for missing coverage:
  - API not covered by any task.
  - DB schema not covered by any task.
- Infer cap: <= 20-30% of total tasks.

Case 2: No tasks, has design
- Goal: full coverage for APIs and DB schema.
- For each API: generate 1 task.
- For each DB schema item: generate:
  - design schema task
  - create migration task
- Features are secondary, used only for naming/grouping.

Case 3: Only features
- Goal: full coverage features.
- For each feature: generate 1-N tasks.
- Feature becomes primary context only in this fallback case.

### Updated Flow
1. Determine primary signal by source type first, then origin:
  - planning source file present -> tasks are PRIMARY for all planning-source task items.
  - design source file present and no planning primary -> api/db are PRIMARY.
  - feature source file present and no planning/design primary -> no primary item set; generate technical design tasks instead.
2. Sort primary items by `signal_origin`: explicit > derived > inferred.
3. Generate tasks based on primary signal:
  - tasks: normalize + deterministic cleanup (no drop).
  - design: 1 task per API, 1 design task per DB schema item, and 1 QC task per API.
  - features: generate technical design tasks first; do not treat feature items as direct primary coverage targets.
4. Ensure full coverage: each primary item maps to at least 1 task.
5. Use secondary signals only for deterministic context cleanup and gap filling.
6. Avoid duplication: do not regenerate existing task intent.

### Coverage Rules
- tasks coverage = 100% when planning sources exist.
- apis coverage = 100% when planning sources do not exist and design sources exist.
- db_schema coverage = 100% when planning sources do not exist and design sources exist.
- features are not primary coverage targets.
- If only feature sources exist, convert feature intent into technical design tasks instead of validating feature coverage directly.

Notes:
- Conditional full coverage is enforced by source type availability.
- Planning input dominates all other sources for task primacy.
- Design input dominates feature input for execution primacy.
- Feature-only input falls back to technical design task generation.
- If both explicit and inferred candidates exist for same intent, keep explicit and discard inferred duplicate.

### Guardrails
- No drop: every primary signal item must have a task mapping.
- No over-generation: limit tasks generated from one signal.
- No duplicate intent: no two tasks with same intent.
- Bounded retries: regenerate loop must stop at max attempts.

### Coverage Validation (Required, Pattern-Based)

```python
def validate_coverage(primary_items, generated_tasks):
  for item in primary_items:
    if not has_related_task(item, generated_tasks):
      raise MissingCoverageError(item)
```

Pattern-based coverage matcher (deterministic):
- Task primary mode:
  - active when a planning source file exists.
  - normalize verb + noun phrase and match by stem overlap.
  - every planning-origin explicit task must be covered by one generated task or retained task.
- Design primary mode:
  - active when no planning source file exists and a design source file exists.
  - API item must map to task text containing method/path token pattern or API title token set.
  - DB schema item must map to task text containing schema/table/migration token set.
  - QC task must be created for each API and must match the API title or endpoint token set.
- Feature fallback mode:
  - no direct primary coverage validation.
  - feature titles are converted into technical design tasks and then validated as task items.

Primary set builder:

```python
def build_primary_items(canonical_model):
  has_planning = has_source_type(canonical_model, "Planning")
  has_design = has_source_type(canonical_model, "Design")
  has_feature = has_source_type(canonical_model, "Requirement")

  if has_planning:
    return {
      "primary_signal_type": "tasks",
      "primary_items": canonical_model.tasks,

    "primary_items": [],

  }
```

### 7.1 Post-Generation Critical Loop

Goal:
- Critique generated tasks after deterministic checks and regenerate only when necessary.

Flow:
1. Run pattern-based coverage validation.
2. Collect issues into `CoverageIssueDTO[]`:
  - ERROR: missing primary coverage, invalid source propagation, hard duplicate intent.
  - WARNING: weak task wording, low-specificity description, soft overlap duplicates.
3. Build `TaskCriticalContextDTO` and call Critical AI (critic role only).
4. Critical AI returns structured critique:
  - keep/regenerate decision
  - prioritized fix list
  - target items requiring rewrite/add/remove
5. Regenerate only when at least one `ERROR` exists or warning score exceeds threshold.
6. Re-run deterministic coverage validation.
7. Stop when pass criteria met or regenerate budget reaches 0.

### 7.2 Deterministic Pattern Checker (Detailed Design)

Design objective:
- Validate generated tasks against canonicalized entities (originating from reconciliation) with deterministic pattern rules before any AI critique.

Input scope (this phase):
- Primary context: `CanonicalModelOutputDTO` (built from `ReconciliationOutputDTO`).
- Task output context: generated `TaskGenerationOutputDTO.tasks[]`.
- Optional trace context: canonical-to-merge lineage (`canonical_id -> cluster_id/source`).

Checker pipeline:
1. Build primary item set from source-of-truth rule:
  - if canonical `tasks[]` exists -> primary = tasks
  - else if canonical `apis[]` or `db_schema[]` exists -> primary = design
  - else -> primary = features
2. Normalize all texts for matching:
  - lowercase copy
  - strip punctuation except `/` for API path checks
  - collapse spaces
  - light stemming for action verbs (implement/implementation, design/designed, migrate/migration)
3. Build task intent index:
  - intent_key from normalized verb + entity tokens
  - source tuple index: `(source_file_name, source_file_type, signal_origin)`
  - endpoint/schema token index for design coverage
4. Match each primary item to candidate tasks with deterministic scoring.
5. Emit `CoverageIssueDTO[]` + `CoverageSummaryDTO`.

Deterministic matching rules:

Rule group A - Primary coverage (hard):
- A primary item is covered when best candidate score >= 0.55.
- If no candidate passes threshold: `ERROR/MISSING_COVERAGE`.

Rule group B - API design coverage:
- API token extraction:
  - method token: GET/POST/PUT/PATCH/DELETE/OPTIONS/HEAD
  - endpoint segments from path (ignore `{id}` placeholders)
- Task must satisfy at least one condition:
  - method exact match + endpoint segment overlap >= 0.5
  - endpoint segment overlap >= 0.7
  - alias/title token overlap >= 0.65
- Else: `ERROR/MISSING_COVERAGE`.

Rule group C - DB schema coverage:
- DB token extraction:
  - table/entity tokens from title/aliases
  - structural keywords: schema, migration, index, constraint, column
- Coverage conditions:
  - at least one schema-design task match (token overlap >= 0.6)
  - and one migration task match when generation mode is design-only
- Missing one required mapping: `ERROR/MISSING_COVERAGE`.

Rule group D - Feature fallback coverage:
- Feature title/alias token overlap with any task >= 0.55.
- If only generic overlap words match (system/module/feature): mark as not covered.

Rule group E - Duplicate intent (hard + soft):
- Build intent signature from normalized action + entity tokens.
- Hard duplicate:
  - exact same intent signature appears >= 2 times
  - emit `ERROR/DUPLICATE_INTENT`
- Soft duplicate:
  - cosine(token_vector) >= 0.9 but not exact signature
  - emit `WARNING/DUPLICATE_INTENT`

Rule group F - Source propagation:
- For each mapped task, verify `source_file_name`, `source_file_type`, `signal_origin` consistency with mapped primary item.
- Missing or conflicting values: `ERROR/SOURCE_MISMATCH`.

Rule group G - Task quality floor:
- Reject weak task names:
  - length < 10 chars
  - verb-only titles (implement/create/design/update/fix with no entity)
- Reject weak descriptions:
  - length < 20 chars
  - missing implementation detail tokens
- Emit `WARNING/WEAK_TASK`.

Scoring model (deterministic):
- score = 0.40 * token_overlap + 0.30 * structural_match + 0.20 * source_match + 0.10 * origin_priority_match
- token_overlap: Jaccard overlap on normalized tokens
- structural_match:
  - API method/path structure match or DB schema/migration structure match
- source_match:
  - 1.0 when same source file/type, else 0.0
- origin_priority_match:
  - explicit=1.0, derived=0.7, inferred=0.4

Issue aggregation policy:
- `ERROR` stops pass immediately but still collect all issues in one run.
- `WARNING` does not fail pass alone unless warning score threshold is exceeded.
- warning score = `2 * duplicate_warning + weak_task_count + invalid_format_count`.
- Regenerate trigger when warning score >= 5.

### 7.5 Technical Design Task Fallback

Goal:
- When only feature sources exist, generate technical design tasks instead of treating feature items as direct primary entities.

Rules:
- Convert each feature into one or more technical design tasks.
- Prefer implementation-oriented wording over business-summary wording.
- Keep source fields from the feature source file.
- If a feature implies API or DB work, emit technical design tasks that name the technical work explicitly.

Examples:
- Feature: login flow -> Technical Design: define auth flow, define token validation, write API contract.
- Feature: dashboard UI -> Technical Design: define screen structure, define component layout, define data binding.

QC Task Requirement for Design Input:
- For each API item, generate a QC task in addition to implementation task(s).
- QC task intent should be one of:
  - write test cases for API
  - execute API test cases
  - verify API response and error handling
- QC tasks inherit the same source_file_name, source_file_type, and signal_origin as the API task they validate.

Post-regenerate requirement:
- Must run the same deterministic checker again.
- If still failing and retry budget remains, call Critical API again with refreshed files.
- If budget exhausted, return best-effort tasks + full issue list.

Pass criteria:
- `ERROR` count = 0
- Primary coverage = 100%
- Duplicate intent count under configured threshold

Regenerate policy:
- Max attempts: 2 (configurable).
- Delta-only regeneration: modify only affected tasks from critique output.
- Preserve valid tasks unchanged.

Critical AI output contract:

```json
{
  "action": "KEEP" | "REGENERATE",
  "severity": "LOW" | "MEDIUM" | "HIGH",
  "summary": "short reason",
  "fixes": [
    {
      "issue_id": "cov-001",
      "operation": "ADD" | "UPDATE" | "REMOVE",
      "target": "task-name-or-signal-id",
      "instruction": "concrete deterministic rewrite instruction"
    }
  ]
}
```

Execution skeleton:

```python
def generate_with_critical_loop(input_payload, max_attempts=2):
  tasks = generate_tasks(input_payload)

  for _ in range(max_attempts + 1):
    issues, coverage = run_pattern_coverage_checks(input_payload, tasks)
    if coverage.error_count == 0 and coverage.primary_coverage == 1.0:
      return tasks, issues

    ctx = build_critical_context(input_payload, tasks, issues, coverage)
    critique = call_critical_ai(ctx)

    if critique["action"] == "KEEP":
      return tasks, issues

    tasks = regenerate_delta(tasks, critique["fixes"])

  return tasks, issues
```

Mental model:
- Task generation maps signal to task.
- Critical AI is a bounded quality gate, not an open-ended generator.

## 8. Core Principles

| Principle | Description |
| --- | --- |
| Maximize recall | Prefer coverage over precision |
| No deletion | Never delete extracted signals |
| Tasks as truth | Treat planning task signals as explicit truth |
| API and DB as truth | Treat API and database signals as execution truth |
| Features as fallback | Treat feature signals as fallback intent |
| No hard skips | Avoid hard skips except where planning tasks already exist |
| Dominance over presence | Use dominance, not mere presence, for classification |
